| **CodeEditor.vue** | Редактор кода с подсветкой синтаксиса | CodeMirror 6, vue-codemirror6 |
| **P5Canvas.vue** | Изолированный iframe для выполнения p5.js скетчей | Iframe sandbox, Blob URL |

|------|----------|----------|--------|

### Поток данных

```
Router (Vue Router)
    │
    ├── EditorPage.vue
    │   ├── CodeEditor.vue ←→ code (v-model)
    │   ├── P5Canvas.vue → addMessage(), mouseMove
    │   ├── ConsoleOutput.vue → messages
    │   ├── ExamplesPanel.vue → loadExample(code)
    │   └── AIChat.vue → suggestCode(code)
    │
    ├── ExplorePage.vue
    │   └── Supabase API → getGallerySketches()
    │
    ├── SharePage.vue
    │   └── Supabase API → createSketch()
    │
    ├── SketchDetailPage.vue
    │   └── Supabase API → getSketchById()
    │
    └── AdminDashboard.vue
        └── Supabase API → getPendingSketches(), approveSketch(), rejectSketch()
```

### Управление состоянием

- **Локальное состояние:** Каждый компонент управляет своим состоянием через `ref()`, `reactive()`
- **Двустороннее связывание:** `v-model` для редактора кода
- **События:** `emit()` для коммуникации между компонентами
- **localStorage:** Сохранение состояния в ExamplesPanel (выбранный пример, шаг, вид), роль пользователя
- **Supabase API:** Функции в `src/composables/useSketches.ts` для работы с БД
- **Composables:** `useAuth`, `useSketches`, `useStorage` для общей логики

### Интерфейсы TypeScript

**Файл:** `src/types/supabase.ts`

```typescript
// Роли пользователей
type UserRole = 'user' | 'moderator' | 'admin'

// Статусы скетчей
type SketchStatus = 'pending' | 'approved' | 'rejected' | 'draft'

// Сложность скетчей
type SketchDifficulty = 'Лёгкая' | 'Средняя' | 'Тяжёлая'

// Действия модерации
type ModerationAction = 'approved' | 'rejected'

// Профиль пользователя
interface Profile {
  id: string
  email: string
  display_name: string | null
  avatar_url: string | null
  bio: string | null
  role: UserRole
  created_at: string
  updated_at: string
}

// Скетч
interface Sketch {
  id: string
  user_id: string
  title: string
  description: string
  code: string
  thumbnail_url: string | null
  tags: string[]
  category: string | null
  difficulty: SketchDifficulty | null
  status: SketchStatus
  rejection_reason: string | null
  views: number
  likes: number
  created_at: string
  updated_at: string
}

// Лог модерации
interface SketchModerationLog {
  id: string
  sketch_id: string
  moderator_id: string
  action: ModerationAction
  comment: string | null
  created_at: string
}
```

### Проверка работы авторизации

**Рабочий процесс:**

1. **Регистрация пользователя:**
   - Пользователь вводит email, пароль и display name
   - Вызывается `supabase.auth.signUp()`
   - Создаётся пользователь в `auth.users`
   - Триггер `on_auth_user_created` автоматически создаёт профиль в `profiles`
   - Профиль загружается в приложение через `loadProfile()`

2. **Вход пользователя:**
   - Пользователь вводит email и пароль
   - Вызывается `supabase.auth.signInWithPassword()`
   - Загружается профиль через `loadProfile()`
   - Обновляется состояние в `useAuth`

3. **Загрузка аватара:**
   - Пользователь выбирает файл изображения
   - Файл загружается в Storage: `user-content/avatars/{user-id}-{timestamp}.ext`
   - Получается публичная ссылка через `getPublicUrl()`
   - Профиль обновляется с новой ссылкой на аватар

4. **Обновление профиля:**
   - Пользователь редактирует display name и bio
   - Вызывается `updateProfile()` с новыми данными
   - Профиль обновляется в БД и в состоянии приложения

5. **Автосохранение сессии:**
   - При загрузке приложения проверяется сессия через `getSession()`
   - Подписка на изменения через `onAuthStateChange()`
   - При изменении состояния загружается профиль пользователя

### Workflow модерации

```
1. Пользователь создаёт скетч
   ↓
2. Скетч сохраняется со статусом 'pending'
   ↓
3. Скетч появляется в админ-панели на модерации
   ↓
4. Модератор просматривает скетч
   ↓
   ├─→ Одобрить → status: 'approved' → В галерею
   │
   └─→ Отклонить → status: 'rejected' → Автор получает причину
```

### Функции модерации

#### Одобрение скетча

```typescript
const { approveSketch } = useSketches()
const result = await approveSketch(
  'sketch-id',
  'moderator-id',
  'Отличная работа!' // опционально
)
```

**Что происходит:**
1. Обновляется статус скетча: `status = 'approved'`
2. Создаётся запись в `sketch_moderation_logs` с действием `approved`
3. Скетч появляется в галерее (`/explore`)

#### Отклонение скетча

```typescript
const { rejectSketch } = useSketches()
const result = await rejectSketch(
  'sketch-id',
  'moderator-id',
  'Код содержит ошибки в строке 5'
)
```

**Что происходит:**
1. Обновляется статус скетча: `status = 'rejected'`
2. В поле `rejection_reason` сохраняется причина
3. Создаётся запись в `sketch_moderation_logs` с действием `rejected`
4. Скетч не появляется в галерее

### RLS политики для модерации

**Для `sketches`:**

```sql
-- Модераторы могут обновлять статус любого скетча
CREATE POLICY "Moderators can update any sketch status"
  ON public.sketches FOR UPDATE
  USING (
    EXISTS (
      SELECT 1 FROM public.profiles
      WHERE id = (SELECT auth.uid())
      AND role IN ('moderator', 'admin')
    )
  );

-- Модераторы могут просматривать все скетчи
CREATE POLICY "Moderators can view all sketches"
  ON public.sketches FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.profiles
      WHERE id = (SELECT auth.uid())
      AND role IN ('moderator', 'admin')
    )
  );
```

**Для `sketch_moderation_logs`:**

```sql
-- Модераторы могут просматривать логи
CREATE POLICY "Moderators can view moderation logs"
  ON public.sketch_moderation_logs FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.profiles
      WHERE id = (SELECT auth.uid())
      AND role IN ('moderator', 'admin')
    )
  );

-- Модераторы могут создавать логи
CREATE POLICY "Moderators can create moderation logs"
  ON public.sketch_moderation_logs FOR INSERT
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM public.profiles
      WHERE id = (SELECT auth.uid())
      AND role IN ('moderator', 'admin')
    )
  );
```
